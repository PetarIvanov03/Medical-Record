========================================================================
МЕДИЦИНСКО ДОСИЕ НА ПАЦИЕНТА - СПЕЦИФИКАЦИЯ НА REST API ЕНДПОЙНТИ
========================================================================
Автор на структурата: Проект по CSCB869 Java Web Services
Версия: 1.0 (Пълна спецификация с включени роли и администраторски права)

Легенда за правата на достъп (Spring Security Roles):
- [ALL]    - Достъпно за всеки (без автентикация)
- [ADMIN]  - Пълен достъп до всички данни и административни функции
- [DOCTOR] - Достъп до данни на пациенти, създаване/редакция на собствени прегледи
- [PATIENT]- Достъп само до собственото медицинско досие и история

------------------------------------------------------------------------
1. AUTH CONTROLLER (Автентикация и Потребители)
------------------------------------------------------------------------
POST   /api/auth/register
       - Достъп: [ALL]
       - Описание: Регистрация на нов потребител (пациент).
       - Request Body: UserRegisterDTO (username, password, name, egn)

POST   /api/auth/login
       - Достъп: [ALL]
       - Описание: Вход в системата. Връща JWT токен с ролята.
       - Request Body: UserLoginDTO (username, password)
       - Response: JwtAuthResponseDTO (token, username, role)

GET    /api/admin/users
       - Достъп: [ADMIN]
       - Описание: Списък с всички потребителски профили в системата.

DELETE /api/admin/users/{id}
       - Достъп: [ADMIN]
       - Описание: Изтриване на потребителски профил и акаунт.

------------------------------------------------------------------------
2. DOCTOR CONTROLLER (Управление на Лекари)
------------------------------------------------------------------------
GET    /api/doctors
       - Достъп: [ADMIN, DOCTOR, PATIENT]
       - Описание: Извежда всички лекари (за избор на личен лекар или специалист).

GET    /api/doctors/{id}
       - Достъп: [ADMIN, DOCTOR, PATIENT]
       - Описание: Детайлна информация за конкретен лекар.

POST   /api/admin/doctors
       - Достъп: [ADMIN]
       - Описание: Създаване на нов лекар и обвързването му с потребителски акаунт.
       - Request Body: DoctorCreateDTO (uin, name, specialtyId, isGp, username, password)

PUT    /api/admin/doctors/{id}
       - Достъп: [ADMIN]
       - Описание: Актуализиране на данните на лекар (смяна на специалност, статус на личен лекар).
       - Request Body: DoctorUpdateDTO

DELETE /api/admin/doctors/{id}
       - Достъп: [ADMIN]
       - Описание: Изтриване на лекар от системата.

------------------------------------------------------------------------
3. PATIENT CONTROLLER (Управление на Пациенти)
------------------------------------------------------------------------
GET    /api/patients
       - Достъп: [ADMIN, DOCTOR]
       - Описание: Списък с всички пациенти в болницата.

GET    /api/patients/{id}
       - Достъп: [ADMIN, DOCTOR]
       - Описание: Профил на конкретен пациент по неговото ID.

GET    /api/patients/my-profile
       - Достъп: [PATIENT]
       - Описание: Пациентът достъпва собствения си профил (взема се от JWT токена).

PUT    /api/patients/my-profile/change-gp
       - Достъп: [PATIENT]
       - Описание: Избор или промяна на личен лекар (gp_id).
       - Request Body: ChangeGpDTO (gpId)

GET    /api/patients/{id}/history
       - Достъп: [ADMIN, DOCTOR]
       - Описание: История на посещенията и прегледите на конкретен пациент.

GET    /api/patients/my-history
       - Достъп: [PATIENT]
       - Описание: Пациентът вижда единствено своята собствена медицинска история.

POST   /api/admin/patients
       - Достъп: [ADMIN]
       - Описание: Ръчно служебно създаване на пациент от администратор.
       - Request Body: PatientCreateDTO

DELETE /api/admin/patients/{id}
       - Достъп: [ADMIN]
       - Описание: Изтриване на пациент от системата.

------------------------------------------------------------------------
4. EXAMINATION CONTROLLER (Прегледи)
------------------------------------------------------------------------
GET    /api/examinations
       - Достъп: [ADMIN]
       - Описание: Списък на всички извършени прегледи в болницата (за одит).

GET    /api/examinations/my-history
       - Достъп: [DOCTOR]
       - Описание: Извежда списък само с прегледите, извършени от текущия лекар.

GET    /api/examinations/{id}
       - Достъп: [ADMIN, DOCTOR, PATIENT]
       - Описание: Преглед на детайли за конкретен преглед (със защита в Service слоя).

POST   /api/examinations
       - Достъп: [DOCTOR]
       - Описание: Въвеждане на нов преглед.
       - Бизнес логика: Автоматично изчислява paid_by_nzok спрямо is_insured на пациента.
       - Request Body: ExaminationRequestDTO (patientId, diagnosisId, treatment, price)

PUT    /api/examinations/{id}
       - Достъп: [DOCTOR]
       - Описание: Редакция на преглед. Валидация: Само лекарят, който го е създал.
       - Request Body: ExaminationRequestDTO

DELETE /api/admin/examinations/{id}
       - Достъп: [ADMIN]
       - Описание: Изтриване/Анулиране на сгрешен преглед.

------------------------------------------------------------------------
5. SICK LEAVE CONTROLLER (Болнични Листове)
------------------------------------------------------------------------
GET    /api/sick-leaves/{id}
       - Достъп: [ADMIN, DOCTOR, PATIENT]
       - Описание: Преглед на конкретен болничен лист.

POST   /api/sick-leaves
       - Достъп: [DOCTOR]
       - Описание: Издаване на болничен към конкретен преглед (Релация 1:1).
       - Request Body: SickLeaveRequestDTO (examinationId, startDate, durationDays)

PUT    /api/sick-leaves/{id}
       - Достъп: [DOCTOR]
       - Описание: Редакция на болничен от лекаря, който го е издал.

DELETE /api/sick-leaves/{id}
       - Достъп: [ADMIN, DOCTOR]
       - Описание: Анулиране на болничен лист.

------------------------------------------------------------------------
6. STATISTICS CONTROLLER (Справки и Статистика - главно за ADMIN)
------------------------------------------------------------------------
GET    /api/statistics/patients-by-diagnosis?diagnosisId={id}
       - Достъп: [ADMIN, DOCTOR]
       - Описание: Списък на всички пациенти с дадена диагноза.

GET    /api/statistics/most-common-diagnosis
       - Достъп: [ADMIN, DOCTOR]
       - Описание: Намиране на най-често срещаната диагноза в системата.

GET    /api/statistics/patients-by-gp?gpId={id}
       - Достъп: [ADMIN, DOCTOR]
       - Описание: Списък с пациенти, записани при даден личен лекар.

GET    /api/statistics/gp-patient-counts
       - Достъп: [ADMIN]
       - Описание: Брой пациенти при всеки личен лекар (Справка за НЗОК).

GET    /api/statistics/doctor-visit-counts
       - Достъп: [ADMIN]
       - Описание: Брой посещения (прегледи) при всеки лекар.

GET    /api/statistics/total-revenue
       - Достъп: [ADMIN]
       - Описание: Обща стойност на платените от пациентите прегледи (където paid_by_nzok = false).

GET    /api/statistics/revenue-by-doctor?doctorId={id}
       - Достъп: [ADMIN]
       - Описание: Стойност на платените от пациентите прегледи за конкретен лекар.

GET    /api/statistics/peak-sick-leave-month
       - Достъп: [ADMIN]
       - Описание: Месец с най-много издадени болнични листове в системата.

GET    /api/statistics/doctors-with-most-sick-leaves
       - Достъп: [ADMIN]
       - Описание: Лекар или лекари с най-много издадени болнични.

GET    /api/statistics/examinations-by-period?startDate=YYYY-MM-DD&endDate=YYYY-MM-DD
       - Достъп: [ADMIN, DOCTOR]
       - Описание: Прегледи, филтрирани по период от време.
       
------------------------------------------------------------------------
7. NOMENCLATURE CONTROLLER (Номенклатури за падащи менюта)
------------------------------------------------------------------------
GET    /api/specialties
       - Достъп: [ADMIN, DOCTOR]
       - Описание: Списък с всички медицински специалности за нуждите на формите.

GET    /api/diagnoses
       - Достъп: [DOCTOR]
       - Описание: Списък с всички налични диагнози за избор по време на преглед.
